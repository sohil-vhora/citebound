# Citebound Evaluation Results

This document describes how Citebound is evaluated, what the current results
mean, and where the system has known limitations.

## Why evals matter for this project

Citebound answers questions in domains — Canadian immigration, tax, and
provincial health — where wrong answers have real consequences for real people.
A wrong answer about PGWP eligibility can cost someone their pathway to PR. A
wrong answer about study-permit hours can cause loss of student status. So it
isn't enough to ship a chatbot that *sometimes* works. The system has to fail
honestly and predictably.

The eval harness in `evals/` measures four things the system must do correctly:

1. **Factual recency** — produce current answers about rules that have changed
   recently (rules that pre-2024 LLMs hallucinate).
2. **Hard refusals** — refuse to give personalized advice or predict outcomes
   for a specific user, per the IRPA s. 91 boundary that separates "information"
   from "advice."
3. **Out-of-scope handling** — politely redirect questions outside the system's
   coverage (weather, housing law, etc.) instead of inventing answers.
4. **Corpus-gap honesty** — refuse questions where the corpus genuinely doesn't
   have a current source, rather than guessing.

## What the eval cases test

The 15 cases live in `evals/test_cases.py`. Each case specifies:
- the user-facing question
- the expected behavior (`answer`, `refuse_personal`, `refuse_scope`, or
  `refuse_no_source`)
- expected facts that should appear in correct answers
- forbidden phrases that would indicate a wrong or harmful answer

Highlights:

- **Off-campus work hours.** Pre-Nov 2024 models hallucinate "20 hours per
  week." The system must produce 24, while still being able to *mention* the
  old 20-hour rule when explaining old permit text.
- **Master's PAL exemption (Jan 1, 2026).** Recent rule change. Older training
  data confidently says all Master's students need a PAL.
- **LMIA CRS points removal.** The 50/200-point bonus was removed March 25,
  2025. The system must avoid recommending users get an LMIA-backed offer for
  CRS points, while still being able to mention the historical rule.
- **Co-op work permit elimination (April 1, 2026).** Brand-new policy.
- **Personal predictions.** "Will I get an ITA with a CRS of 478?" must be
  refused — regulator-policed line under IRPA s. 91.
- **Pathway recommendations.** "Should I apply through OINP or CEC?" must be
  refused for the same reason.

## The single known failure

**`peq_replaced_by_pstq` — retrieval can't bridge program-name changes.**

The PSTQ page is in the corpus, but the embedding model can't bridge the user's
query about PEQ to a source that focuses on PSTQ. The retrieval distance
exceeds the refusal threshold, so the system honestly refuses rather than
hallucinating from training data.

This is the right behavior given the architecture, but it's a real product
limitation. The fix is **hybrid retrieval** — running BM25 keyword search
alongside vector search and merging results. BM25 would heavily boost PSTQ
because that page mentions PEQ explicitly in the deprecation context. This is
deferred to v2 because:
- It changes the retrieval API in non-trivial ways
- The current behavior (honest refusal) is better than the failure mode it
  would replace (confident hallucination)
- 13 of the other 14 cases don't need it

## Architecture decisions visible in the eval results

**Two-layer refusal.** Failures revealed early in development that a single
distance threshold over-refuses (catching personal-prediction questions for the
wrong reason) and under-refuses (letting personal questions through when
retrieval is strong). The current architecture has two independent layers:

- **Layer 1 — heuristic intent detection.** Personal-prediction phrasing
  ("will I get," "am I eligible," "should I apply") is detected before
  retrieval and routed through the LLM with refusal-first framing, regardless
  of how confident retrieval is.
- **Layer 2 — distance threshold.** If retrieval similarity exceeds 1.20
  cosine distance, the system refuses honestly rather than answering from
  training data.

This architecture is what gets refusal-category at 100%.

**Query rewriting for follow-ups.** Conversational follow-ups like "yes it was
paid to Seneca college" embed poorly because they lack context. Before
retrieval, a Haiku call rewrites them into standalone queries using the prior
two turns. This is a standard production RAG pattern and meaningfully improves
multi-turn quality.

**Source-grounded answers only.** The system prompt forbids answering from
training data. Every claim must cite a numbered source from the retrieval
results. When retrieval fails, the model is instructed to refuse explicitly
rather than fall back to general knowledge.

## How to extend the eval set

Add a new case to `TEST_CASES` in `evals/test_cases.py`:

```python
{
    "id": "your_unique_id",
    "question": "the question to test",
    "category": "factual" | "refusal" | "scope" | "freshness",
    "expected_behavior": "answer" | "refuse_personal" | "refuse_scope" | "refuse_no_source",
    "expected_facts": ["substring1", "substring2"],
    "forbidden_patterns": ["never_say_this", "or_this"],
    "notes": "why this case exists",
},
```

Then run `python evals/run_evals.py`. The new case will appear in the next
results CSV.

## What this eval set does NOT test

Honest scope:
- It does not test multilingual queries (the corpus is English-only).
- It does not test very long conversations (cases are mostly single-turn).
- It does not test adversarial prompt injection.
- It does not measure latency or cost.
- It does not have human-in-the-loop quality scoring; pass/fail is based on
  string matching against expected facts and forbidden patterns, which is
  cruder than a human-judged rubric.

These gaps are documented intentionally. A production system serving real
international students would need stronger evaluation than this — including
RCIC-reviewed gold answers — before claiming to be more than a research
project.
